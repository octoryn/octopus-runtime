# 安全策略

## 报告漏洞

请**不要**为安全漏洞公开提交 issue。

请通过 GitHub Security Advisories(仓库 Security 标签页中的 "Report a vulnerability")或发送邮件至
**security@octopusos.ai** 私下报告。请附上问题描述、复现步骤和影响范围。我们会争取在几个工作日内予以确认。

## 范围说明

Octopus Runtime 治理"工作如何从观察走向行动",因此以下几处在设计上就与安全强相关:

- **结构性安全是核心保证** —— 连接器中有副作用的 `execute` 必须仅在 Autonomous(自主)路径上、
  或在 Draft(草稿)审批通过之后才可达;有效自主级别始终为 `min(requested, 所有策略)`。任何让效果在
  高于其受治理自主级别的情况下执行的缺陷,都属于安全问题,请予以报告。
- **连接器是通往外部世界的信任边界。** 内置的 HTTP 连接器会请求工作流所规划的任意 URL;若该 URL 的任何
  部分来自不可信输入,请加以约束(在规划器中或通过策略限制 host/scheme),以防 SSRF。运行时治理的是
  *是否*发起请求,而非请求*去向何方*。
- **密钥**通过 `SecretProvider` 端口提供,由连接器在执行时读取;它们绝不会被持久化进运行记录或审计中。
- **持久化存储**会把 运行/审批/审计 数据以 JSON 或 SQLite 的形式写入你所指向的主机 —— 请保护好该数据目录 /
  数据库文件。

## 支持的版本

本项目处于 1.0 之前阶段;仅最新版本会获得修复。
